# ADR 0001 — Java 21 as the platform baseline

**Status:** Accepted (2026-07)

## Context

Atlas is deliberately a Java-first RAG platform. Most RAG tooling is prototyped in Python and
rebuilt for the JVM when enterprise requirements (type safety, existing Spring estates, ops
maturity) demand it. Atlas targets that rebuild-side gap directly, so the JVM choice is the
project's premise, not an implementation detail. The remaining question is which Java version.

## Decision

Java 21 (LTS) is the baseline for `atlas-core`. No support for earlier versions.

## Rationale

- **LTS with modern language features.** Records for DTOs (citations, parsed documents), sealed
  interfaces for parser/status hierarchies, pattern matching in ingestion state handling, and
  virtual threads for I/O-heavy work (embedding calls, LLM calls) without reactive complexity.
- **Ecosystem alignment.** Spring Boot 3.x requires Java 17+; 21 is the LTS the Spring AI and
  Boot 3.4+ generation is tested against.
- **Enterprise credibility.** The target audience for Atlas (and its FDE playbooks) runs LTS
  releases. 21 is the conservative-modern choice: new enough to be current, old enough to be
  deployable in a bank.

## Consequences

- Virtual threads are available for the embedding/generation call paths, keeping the concurrency
  model simple (blocking style) rather than WebFlux. This is a deliberate simplicity trade.
- Contributors need JDK 21; the Maven wrapper and CI pin it, so drift is caught mechanically.
- When Java 25 LTS adoption matures, migration should be low-cost since no preview features are
  used.
